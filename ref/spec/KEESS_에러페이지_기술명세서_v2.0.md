## 2. 대응 범위: 상황별 매트릭스

| # | 상황 | 예시 | 대응 화면 | 담당 파일 |
|---|---|---|---|---|
| 1 | 존재하지 않는 경로 접근 | `/abc`, `/ax-ai2`, 구버전 URL | 404 | `app/not-found.tsx` |
| 2 | 코드에서 수동 콘텐츠 판정 | `notFound()` 호출 | 404 | `app/not-found.tsx` |
| 3 | 페이지 렌더링 중 오류 | 컴포넌트 예외, 데이터 처리 실패 | 500 | `app/error.tsx` |
| 4 | 루트 레이아웃 자체 오류 | 공통 레이아웃 렌더 실패 | 500 (독립형) | `app/global-error.tsx` |
| 5 | 상담 폼 API 실패 | 제출 시 서버·네트워크 오류 | **500으로 이동하지 않음.** 폼 내부 인라인 오류 메시지로 처리 | 기존 폼 컴포넌트 |
| 6 | 호스팅 플랫폼 장애 | Vercel 자체 장애·과부하(503 등) | 플랫폼 기본 화면. 애플리케이션 통제 범위 밖 | 대응 불가 (인지만) |
| 7 | 점검·일시 중단 안내 | 계획된 유지보수 | 본 건 범위 외. 필요 시 2차 검토 | 해당 없음 |

**핵심 판단 2가지**

- **5번이 가장 중요한 경계점**: 상담 폼 제출 실패는 사용자가 입력을 이미 마친 상태이므로, 500 페이지로 이동시키면 입력 내용이 사라져 리드를 확실히 잃는다. 반드시 폼 안에서 인라인 처리하고 입력값을 유지한다.
  - **구현 확인 결과**: 폼은 클라이언트 상태 기반이라 입력값이 유지되며, 이벤트 핸들러 오류는 React error boundary로 전파되지 않으므로 500 이동 경로가 구조적으로 존재하지 않는다. (충족)
- **6번은 인지 사항**: 플랫폼 자체 장애 시에는 어떤 화면도 뜨지 않는다. 대응 범위가 아님을 팀 내 공유해 두면 장애 시 혼선이 없다.

## 3. 파일별 구현 명세

### 3-1. `app/not-found.tsx` (404)

| 항목 | 기준 |
|---|---|
| 표시 조건 | 존재하지 않는 라우트 접근, `notFound()` 호출 |
| HTTP 상태코드 | 반드시 404 반환 (**200 반환 금지**: soft 404는 검색엔진이 오류 화면을 정상 페이지로 오인) |
| 레이아웃 | 공통 GNB(solid)·푸터 포함 |
| 외부 리소스 | 허용 (Pretendard 폰트, 기존 전역 CSS 재사용) |
| 링크 제약 | 화면 내 모든 링크는 실제 5개 라우트(`/`, `/ax-ai`, `/leadership`, `/hrd`, `/content`)와 상담 앵커(`/#inq`)만 사용. 에러에서 에러로 가는 순환 금지 |

### 3-2. `app/error.tsx` (라우트 오류)

| 항목 | 기준 |
|---|---|
| 필수 규약 | 첫 줄 `'use client'`, `error`·`reset` props 수신 |
| 다시 시도 | 주 CTA에서 `reset()` 호출 (새로고침 없이 재렌더) |
| 레이아웃 | GNB·푸터 미포함 독립 화면 |
| 외부 리소스 | 이미지 금지. 폰트·스타일은 컴포넌트 내부에서 해결 |
| 로깅 | `error` 객체를 콘솔에 기록. 사용자 화면에는 오류 원인 미노출 |

### 3-3. `app/global-error.tsx` (전역 오류)

| 항목 | 기준 |
|---|---|
| 필수 규약 | `'use client'` 선언, `html`·`body` 태그를 파일 안에서 직접 포함 |
| 독립성 | 전역 CSS·웹폰트·이미지 일절 미참조. 시스템 폰트 스택, 인라인 스타일만 사용 |
| 화면 구성 | `error.tsx`와 동일 내용(독립 동작용 하드코딩 복사만 예외 허용) |

## 5. 금지 사항

| # | 금지 항목 | 이유 |
|---|---|---|
| 1 | 404에서 HTTP 200 반환 (soft 404) | 검색엔진이 오류 화면을 정상 페이지로 오인 |
| 2 | 에러 페이지 내 링크가 없는 라우트를 가리킴 | 에러에서 에러로 가는 순환 |
| 3 | `global-error`에서 외부 리소스 참조 | 장애 상황에서 화면 자체가 안 뜸 |
| 4 | GA·애널리틱스 코드 삽입 | GA는 오픈 후 적용 방침. `page_group` 값 `error`만 추후 예약 |
| 5 | 오류 원인(스택·메시지 원문) 화면 노출 | 보안·신뢰도 문제 |
| 6 | 폼 제출 실패 시 500 페이지로 이동 | 입력값 손실로 리드 확정 손실 |
| 7 | 신규 색상·신규 폰트 임의 추가 | 디자인 일관성 (global-error의 독립 동작용 하드코딩 복사만 예외) |
| 8 | 에러 페이지에 대형 실사 이미지 도입 | 복귀 속도 저하·시선 분산. 500은 외부 리소스 미참조 원칙과 충돌 |

## 8. 확인 방법

- **404**: 운영 URL 뒤에 없는 경로를 붙여 접근 (`/abc` 등). 상태코드는 DevTools Network 탭 또는 `curl -I`로 확인.
- **500**: 실제 런타임 오류가 있어야 렌더되므로 URL만으로는 재현 불가. **Preview 브랜치에 임시 throw 라우트를 두고 Preview URL에서 확인** 후 브랜치를 삭제하는 방식을 권장(운영 무영향). 500 화면은 하이드레이션 후 클라이언트에서 그려지므로 소스 보기·`curl`이 아닌 **브라우저 화면**으로 확인한다.
- **global-error**: 루트 레이아웃 실패 상황이라 URL 재현 불가. 로컬에서 루트 레이아웃에 임시 `throw`를 넣어 확인 후 되돌린다.
